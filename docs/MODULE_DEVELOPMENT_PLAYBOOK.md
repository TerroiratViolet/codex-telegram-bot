# Bot 模块开发手册

这份手册用于把一次临时需求变成可重复、可审查、可发布的 Bot 模块开发流程。未来开发
新的命令、按钮、图片、AI 分析、管理员界面或用户流程时，Codex 必须先读取本文件，
再要求你补全“模块需求填写表”。

本项目的目标不是让 Codex 随手改代码，而是让你可以用手机上的自然语言描述需求，
Codex 仍然能按工程流程完成：理解需求、拆清边界、修改代码、补测试、创建 PR、等待 CI、
合并后由 Railway 发布，并在 Telegram 中验收。

## 1. 核心需求提炼

你真正需要的不是“写一个 Telegram Bot”，而是一套可以反复扩展的工作系统：

1. 初级使用者也能提出需求，不需要手动打开记事本改代码。
2. Codex 能把自然语言需求转成可审查的 Pull Request。
3. 每个功能都有最小可验收目标，而不是一次做成无法判断对错的大工程。
4. Bot 在 Telegram 中能真实响应，且不会扰乱群聊。
5. 管理员与普通用户看到的界面、权限和信息边界必须清楚。
6. 所有 Token、API Key、`.env`、用户隐私内容都不能进入代码、日志、提交或 PR。
7. 生产发布必须经过 `main`、CI 和 Railway，不能从功能分支直接部署。
8. 每次问题修复都要记录底层原因，并增加能预防复发的测试、文档或运行检查。

## 2. 最小成功标准

任何新模块至少要做到下面这些，才算可以交付：

- Codex 可以使用：需求已写成清晰模块说明，缺失信息已向你确认。
- 项目可以跑：本地依赖、启动方式和配置变量没有被破坏。
- Bot 可以响应：至少有一个 Telegram 命令、按钮或消息路径能被触发并返回预期内容。
- 变更可以审查：代码在功能分支，PR 展示清楚改了哪些文件、为什么改、怎么测。
- 服务可以发布：CI 通过，合并到 `main` 后 Railway 可以按既有流程自动部署。
- 用户边界清楚：普通用户、管理员、群聊和私聊分别能看到什么，不会混在一起。
- 验收可重复：有本地测试、smoke test、PR CI 和生产 Telegram 验收步骤。

如果一个想法太大，先切成最小模块。先让一条路径从输入到输出闭环，再升级。

## 4. Codex 标准执行流程

### 4.1 理解与澄清

1. 读取 `AGENTS.md`、`README.md`、本手册和相关源码。
2. 对照“模块需求填写表”找缺口。
3. 如果缺口会导致权限、隐私、部署或用户体验出错，先向你提问。
4. 如果缺口不影响最小目标，写明合理假设并继续。
5. 把需求拆成“主要问题”和“次要问题”，先完成主要问题。

### 4.2 架构判断

1. Telegram 接收、按钮、回调和发送消息放在 `schedule_bot/handlers.py`。
2. 回复文案、状态判断和业务规则放在 `schedule_bot/responses.py` 或独立可测试模块。
3. 配置从环境变量读取，不能把密钥写入代码。
4. 默认不加数据库。只有重启后必须保存的数据，才进入数据库设计。
5. 长轮询期间保持一个生产 Replica。
6. 管理员和普通用户的可见内容必须分开。
7. 群聊中的内容必须克制，敏感或完整分析优先走管理员私聊。
8. 不要注册全量斜杠命令兜底，例如 `MessageHandler(filters.COMMAND, ...)`。
9. 群组里的普通斜杠命令默认必须回复 Bot 的消息才处理，避免读取群里其他服务的
   `/command`。若模块需要例外，例如 `/tarot` 必须回复目标用户，必须在需求表和测试中
   明确写出触发边界。

### 4.3 实现与测试

1. 从 `main` 创建 `codex/` 功能分支。
2. 修改最小必要文件，不做无关重构。
3. 每个行为变化都补测试。
4. 如果是图片、提示词或流程规则，也要用文档和测试固定关键约束。
5. 本地运行：

```powershell
.\.venv\Scripts\python.exe -m ruff check .
.\.venv\Scripts\python.exe -m pytest
.\.venv\Scripts\python.exe -m schedule_bot --smoke-test "/start"
```

6. 若涉及真实 Telegram 行为，给出手动验收脚本。

### 4.4 审查与发布

1. 只提交相关文件，不提交 `.env`、截图附件、临时文件或密钥。
2. 创建 Pull Request，写清：
   - 改了什么。
   - 为什么这样改。
   - 风险在哪里。
   - 怎么测试。
   - 是否需要 Railway Variables。
3. 等 GitHub Actions 通过。
4. 合并到 `main` 后由 Railway 自动发布。
5. 在 Telegram 用真实命令验收。
6. 如果线上出错，记录底层原因，并补预防性测试或运行检查。

## 5. 边界与注意事项

- 不要让 Bot 重复群里的普通打字内容，除非模块明确要求。
- 不要让 Bot 读取或回复群里所有带 `/` 的消息；未知斜杠命令应静默忽略。
- 不要把常见命令做成群聊直接触发。普通命令在群里应要求用户回复 Bot，或仅限私聊。
- 不要记录完整 Telegram update、塔罗回答、心理分析、用户隐私或外部 AI 输出。
- 不要在聊天、代码、测试、日志、PR 中暴露 Token、API Key 或 `.env`。
- 不要让普通群成员看到管理员私聊内容。
- 不要让管理员专用分析自动公开到群里。
- 不要让 AI 输出被描述成诊断、治疗或确定预测。
- 不要为了一个简单模块提前引入数据库、后台任务系统或复杂框架。
- 不要从功能分支直接部署生产。
- 不要在 CI 失败时合并。
- 不要把“图片看起来差不多”当成通过；必须检查用户明确指出的视觉禁区。

## 6. 互动样本

### 6.1 你给 Codex 的完整需求样本

```text
请阅读 AGENTS.md、README.md 和 docs/MODULE_DEVELOPMENT_PLAYBOOK.md。

我要开发一个“抽签提醒”模块。
普通用户在群里发送 /draw，Bot 随机返回一句简短提示。
管理员可以私聊 /draw_stats 查看当天触发次数。
普通用户不能看统计。
不需要数据库，重启后统计清零可以接受。
不要复述其他群聊文字。
请补测试，运行全部验收命令，创建 PR，不要直接部署。
```

### 6.2 Codex 应该反问的样本

```text
我会按模块开发手册来做。现在还缺 4 个会影响实现的字段：

1. /draw 是否允许私聊使用，还是只允许群聊？
2. 抽签文案是固定内置，还是以后希望你能配置？
3. 当普通用户私聊 /draw_stats 时，要静默忽略还是回复“仅管理员可用”？
4. Railway 发布后，你希望用哪三个 Telegram 操作验收？
```

### 6.3 Codex 完成后的交付样本

```text
已完成“抽签提醒”模块，并开 PR。

行为：
- 群聊 /draw 返回随机提示。
- 管理员私聊 /draw_stats 查看当天内存统计。
- 普通用户不能访问统计。

验收：
- Ruff passed
- Pytest passed
- Smoke test passed
- CI passed

部署影响：
- 不需要新增 Railway Variables。
- 合并 main 后 Railway 自动发布。
```

### 6.4 线上问题复盘样本

```text
问题：管理员没有收到分析。

底层原因：
Gemini 在正式分析请求中返回 503，但 /llmcheck 随后成功，说明配置正确，属于临时服务错误。

修复：
- 对 500/502/503/504 和超时增加短间隔重试。
- 错误消息只发给管理员。
- 不记录用户 A/B/C 原文。

预防：
- 新增测试覆盖临时错误重试。
- 文档记录故障判断方式。
```

## 7. 后续升级路径

### 阶段 1：单路径闭环

目标是让模块能从一个命令或按钮开始，完成一次可见响应。数据可以存在内存里，功能先小。

适合：
- `/about`
- `/ping`
- 简单抽签
- 一张图加一个按钮

### 阶段 2：多步骤状态机

目标是让用户按步骤完成输入，Bot 能识别当前会话状态、超时和取消。

适合：
- 塔罗 A/B/C 回答
- 报名流程
- 问卷
- 管理员审核

### 阶段 3：外部 AI 或第三方 API

目标是把用户授权后的内容发送给外部服务，并处理失败、重试、权限和费用。

适合：
- Gemini 分析
- 图片生成
- 外部搜索
- 支付或订阅状态查询

### 阶段 4：持久化与后台任务

只有当数据必须跨重启保存，或需要定时提醒时，才进入这一阶段。

适合：
- 用户设置
- 历史记录
- 定时提醒
- PostgreSQL
- 任务队列

不要跳阶段。先证明最小路径能跑，再让系统长出更复杂的部分。
